Running scenario: RescueBot delivery

RescueBot is in a small facility. A medical kit must be delivered to a patient room.

Current situation

  • RescueBot starts in \(Room11\).
  • The \(MedKit\) is in \(Room21\).
  • The patient room is \(Room31\).
  • The rooms are connected in a corridor.

Planning question
Find a sequence of actions that makes
\[ Delivered(MedKit, Room31) \]

true.

Available action types
\[ Move, PickUp, Drop. \]

Planning as Symbolic Search

Planning Problem
A planning problem asks for a sequence of actions that transforms an initial state into a state satisfying the goal.

\[ S_0 \to^{a_1} S_1 \to^{a_2} \ldots \to^{a_k} S_k, S_k \models Goal \]

Classical Planning Problem
A classical planning problem assumes a finite state space, full observability, deterministic actions, and a world that changes only through the agent’s actions.
Assumption Meaning
Finite State Space There are only finitely many symbolic states.
Full observability The agent knows which state it is in.
Deterministic actions Each action has one predictable outcome.
Static world Nothing changes unless the agent acts.
Goal achievement The task is to reach a state satisfying the goal.

Why start here?
These assumptions are strong, but they give a clean foundation. Later planning models can relax them to handle uncertainty, sensing, time, costs, and other agents.

What the assumptions buy us
Under classical planning assumptions, every step can be checked: current state + applicable action \(\rightarrow\) next state.

States and Goals

State
A state is a set of ground facts describing what is true in the world at a particular time.
Ground Fact
A ground fact is a predicate applied to concrete objects, with no variables.
  • Ground fact: \(At(RescueBot, Room11)\).
  • Not ground fact: \(At(r, x)\).

The state lists the facts that are true now. A relevant ground fact not listed in the state is treated as false.

Goal
A goal is a condition that must be true in the final state.

A goal says what the final state should satisfy. It does not say how to reach that state.

STRIPS-Style Actions

A STRIPS-style action schema describes an action using a name, parameters, preconditions, add effects, and delete effects.
\[ Action(Move(r, x, y)) \]

Component Role Example
Name The action type. \(Move\)
Parameters Variables used by the action schema. \(r\), \(x\), \(y\)
Preconditions Facts must be satisfied before the action was done. \(Robot(r) \land At(r, x) \land Connect(x, y)\)
Add effects Facts made true after the condition. \(At(r, y)\)
Delete effects Facts made false after the condition. \(At(r, x)\)

Core intuition
Preconditions decide whether an action can happen. Effects describe how the state changes.

Grounding an Action Schema

Grounding an action schema means replacing its variables with concrete objects to obtain a specific action.

Check applicability:
\[ Move(RescueBot, Room11, Room21) \]

Since all preconditions hold in \(S_0\), the action is applicable.

Applying an Action

To apply an action, remove its delete effects from the state and add its add effects.
\[ S' = (S - Del(a)) \cup Add(a) \]

Forward Planning

Forward Planning
Forward planning starts from the initial state and searches forward by applying applicable actions until a goal state is reached.

Backward Planning

Backward Planning
Backward planning starts from the goal and asks which actions could have made the goal true.

Bidirectional Search

Start forward planning and backward planning at the same time, trying to meet in the middle.

From STRIPS tp PDDL

PDDL is a planning domain description language for specifying planning domains and planning problems.

PDDL is a structured way to write the same planning task components we have studied: states, goals, objects, and actions.

File Specifies
Domain File Predicates, action schemas, preconditions, and effects
Problem File Objects, initial state, and goal condition

PDDL-style view of the RescueBot domain

PDDL-style view of the RescueBot problem

The domain says what actions are possible in general. The problem says what specific task we are solving.

End of the symbolic AI block
Search explores possibilities.
Logic represents knowledge.
Planning uses symbolic knowledge to act.
