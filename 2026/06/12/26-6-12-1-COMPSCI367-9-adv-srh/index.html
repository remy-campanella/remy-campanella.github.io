From Single-Agent to Adversial Search

Single Agent Search Adversial Search
Action One agent choose a sequence of actions to reach the goal. Two players alternate moves. MAX tries to maximize the outcome; MIN tries to minimise it.
Objective Find a path that leads to the goal according to a utility function. Choose moves that miximise your outcome assuming the opponent plays optimally.

From Search to Games

Game Specification
\[ (S, A, s_0, F, \mathsf{Player}, \mathsf{Action}, \mathsf{Result}, \mathsf{Utility}) \]

Symbol Meaning
\(S\) states
\(A\) actions
\(s_0\) initial state
\(F \subseteq S\) terminate state
\(\mathsf{Player}(s)\) player to move at state \(s\)
\(\mathsf{Action}(s)\) legal action at \(s\)
\(\mathsf{Result}(s, a)\) successor state after action \(a\)
\(\mathsf{Utility}(s)\) terminal utility for MAX

Zero-sum convention
\[ \mathsf{Utility}(s, \text{MIN}) = -\mathsf{Utility}(s, \text{MAX}) \]

We therefore write terminal utilitiesfrom MAX’s perspective
\[ \mathsf{Utility}(s) = \mathsf{Utility}(s, \text{MAX}) \]

Running Example: Tic-Tac-Toe

Minimax

Start with the simplest terminal utilities
\[ \mathsf{Utility}(s) \in {+1, -1} \]

Bottom-up backup
\[ V(\text{left MIN}) = \min(+1, -1) = -1 \\ V(\text{right MIN}) = \min(+1, +1) = +1 \\ V(\text{root}) = \max(-1, +1) = +1 \]

Backup rule

  • At a MAX node: take the maximum child value.
  • At a MIN node: take the minimum child value.

Worked example: Minimax Values

Recursive minimax value
\[ V(s) = \begin{cases} \mathsf{Utility}(s), &s \in F \\ \max_{a \in \mathsf{Action}(s)}V(\mathsf{Result}(s, a)), &\mathsf{Player(s)} = \text{MAX} \\ \min_{a \in \mathsf{Action}(s)}V(\mathsf{Result}(s, a)), &\mathsf{Player}(s) = \text{MIN} \end{cases} \]

Decision rule at MAX state
\[ a^* \in \arg \max_{a \in \mathsf{Action}(s)}V(\mathsf{Result}(s, a)) \]

Minimax backs up values from leaves to the current state: MAX chooses the largest successor value, while MIN chooses the smallest successor value.

Minimax Pseudocode

Cost of full minimax
Let \(b\) be the branching factor and \(m\) te maximum depth.
\[ 1 + b + b^2 + \ldots + b^m = O(b^m) \]

A depth-first implementation uses
\[ O(bm) \]

space.

Depth-Limited Minimax

Why stop early?
Full minimax explores the whole tree. In large games this is usually impossible.

So practical game agents stop at a cutoff depth \(d\) or when time runs out.

Two extra components

  • Cutoff test: decides when to stop expanding.
  • Evaluation function: estimates a non-terminal state.

\[ Eval(s) \approx \text{true utility of } s \text{ for MAX} \]

Depth-limited value
\[ V_d(s) = \begin{cases} \mathsf{Utility}(s), &s \in F \\ \mathsf{Eval}(s), &d = 0 \\ \max_{a \in \mathsf{Action}(s)}V_{d - 1}(\mathsf{Result}(s, a)), &\mathsf{Player(s)} = \text{MAX} \\ \min_{a \in \mathsf{Action}(s)}V_{d - 1}(\mathsf{Result}(s, a)), &\mathsf{Player}(s) = \text{MIN} \end{cases} \]

Good evaluation functions
They should be fast, larger for better MAX positions, and smaller for better MIN positions.

Alpha-Beta Pruning

Minimax does not always need the exact value of every branch.

Sometimes it is enough to know that a branch is already worse than an option we have found before.

Existing option
Suppose the left branch has already been evaluated: \(V(B) = 3\).

At the root, MAX already has an option worth \(\alpha = 3\).

Alpha-beta pruning is minimax with bounds: stop searching a branch once it cannot affect the final minimax choice.
